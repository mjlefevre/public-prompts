# Role-Based Classification and Tagging Prompts

## QA (Quality Assurance)

1. "Classify these defects [defects] by severity, priority, component, root-cause category, and release-blocking status."
2. "Tag these test scenarios [scenarios] as positive, negative, edge case, boundary, regression, smoke, or exploratory."
3. "Classify these QA risks [risks] by likelihood, impact, detectability, and mitigation urgency."
4. "Tag these requirements [requirements] as clear, ambiguous, missing acceptance criteria, not testable, or needs stakeholder review."
5. "Classify these failed tests [tests] as product defect, test issue, environment issue, data issue, or flaky test."
6. "Tag these bug reports [reports] by quality level: complete, missing reproduction, missing expected result, missing evidence, or duplicate risk."
7. "Classify these release checklist items [items] as must-have, should-have, optional, blocked, or sign-off required."
8. "Tag these production issues [issues] by customer impact, recurrence risk, affected workflow, and test coverage gap."
9. "Classify these automation candidates [candidates] by value, stability, complexity, maintenance cost, and priority."
10. "Tag these QA notes [notes] into findings, questions, assumptions, risks, and action items."

## Business Analysts

1. "Classify these requirements [requirements] as business, functional, non-functional, reporting, data, integration, or compliance."
2. "Tag these stakeholder comments [comments] by theme, sentiment, urgency, decision needed, and affected process."
3. "Classify these business rules [rules] by domain, trigger, exception type, owner, and validation need."
4. "Tag these user stories [stories] as ready, needs acceptance criteria, too large, dependency unclear, or blocked."
5. "Classify these process pain points [pain points] by root cause: people, process, technology, data, policy, or training."
6. "Tag these open questions [questions] by workstream, decision owner, urgency, and dependency."
7. "Classify these capabilities [capabilities] as core, supporting, differentiating, compliance-driven, or future opportunity."
8. "Tag these RFP requirements [requirements] as compliant, partially compliant, non-compliant, unclear, or evidence needed."
9. "Classify these risks [risks] by business impact, operational impact, technical dependency, and change-management concern."
10. "Tag these workshop notes [notes] into decisions, assumptions, objections, dependencies, and next actions."

## Developers

1. "Classify these code issues [issues] as bug, tech debt, refactor, performance, security, reliability, or documentation."
2. "Tag these log entries [logs] by severity, service, failure type, request flow, and likely owner."
3. "Classify these API errors [errors] as validation, authentication, authorization, rate limit, dependency, timeout, or server error."
4. "Tag these pull request comments [comments] as blocking, suggestion, question, nit, security concern, or follow-up."
5. "Classify these test failures [failures] as unit, integration, contract, end-to-end, environment, data, or flaky."
6. "Tag these backlog items [items] by module, complexity, risk, dependency, and estimated effort."
7. "Classify these dependencies [dependencies] by runtime criticality, security risk, update urgency, and ownership."
8. "Tag these database changes [changes] as additive, breaking, migration required, backfill required, index change, or rollback risk."
9. "Classify these TODO comments [todos] as cleanup, missing test, design debt, bug risk, docs, or unknown."
10. "Tag these incidents [incidents] by trigger, affected component, customer impact, detection source, and prevention theme."

## Senior Developers

1. "Classify these architecture risks [risks] by scalability, reliability, security, operability, cost, and organizational ownership."
2. "Tag these technical debt items [items] by customer impact, engineering drag, operational risk, architectural constraint, and urgency."
3. "Classify these RFC comments [comments] as objection, clarification, alternative proposal, risk, implementation detail, or approval."
4. "Tag these system dependencies [dependencies] by criticality, SLA, owner, failure mode, and migration complexity."
5. "Classify these initiatives [initiatives] as platform investment, product enablement, reliability, compliance, cost reduction, or productivity."
6. "Tag these incident action items [items] as prevention, detection, response, documentation, training, or ownership clarification."
7. "Classify these migration tasks [tasks] by phase, reversibility, dependency, blast radius, and rollout risk."
8. "Tag these architecture review findings [findings] by severity, affected quality attribute, required decision, and remediation owner."
9. "Classify these engineering metrics [metrics] as delivery, quality, reliability, productivity, cost, or team-health indicators."
10. "Tag these cross-team blockers [blockers] by team, dependency type, decision needed, urgency, and escalation path."

## Product People

1. "Classify these customer feedback items [items] by theme, segment, sentiment, severity, frequency, and product area."
2. "Tag these feature requests [requests] as retention, activation, revenue, usability, compliance, integration, or support-reduction opportunities."
3. "Classify these roadmap candidates [candidates] by strategic fit, user value, confidence, effort, and risk."
4. "Tag these user interview notes [notes] into needs, pain points, workarounds, objections, quotes, and product ideas."
5. "Classify these metrics [metrics] as acquisition, activation, engagement, retention, monetization, satisfaction, or guardrail metrics."
6. "Tag these experiments [experiments] by hypothesis, funnel stage, confidence, effort, risk, and decision type."
7. "Classify these launch tasks [tasks] by function: product, engineering, design, marketing, sales, support, legal, or analytics."
8. "Tag these churn reasons [reasons] by controllability, product area, customer segment, severity, and evidence strength."
9. "Classify these competitor claims [claims] by feature, positioning, pricing, proof point, threat level, and response needed."
10. "Tag these backlog items [items] using RICE or WSJF fields and flag missing information needed for scoring."
