# Role-Based Extraction and Structuring Prompts

## QA (Quality Assurance)

1. "Extract all testable requirements from this document [doc] and structure them as a table with requirement, test idea, priority, and ambiguity."
2. "Pull defects from this testing transcript [transcript] and organize them by severity, feature area, reproduction status, and owner."
3. "Extract action items from these QA meeting notes [notes] with owner, due date, dependency, and status."
4. "Convert this bug report text [text] into structured fields: title, environment, steps, expected result, actual result, evidence, and impact."
5. "Extract regression test candidates from this release description [description] and group them by risk area."
6. "Turn these exploratory testing notes [notes] into a structured summary of coverage, findings, questions, and follow-ups."
7. "Extract open QA risks from this status update [update] and rank them by likelihood, impact, and mitigation."
8. "Parse this defect list [list] into a table with ID, summary, severity, priority, component, status, and blocker flag."
9. "Extract acceptance criteria from this user story [story] and separate explicit criteria from inferred criteria."
10. "Convert this test execution log [log] into passed, failed, blocked, skipped, and needs-investigation categories."

## Business Analysts

1. "Extract business rules from this requirements document [doc] and structure them by rule, condition, exception, owner, and source."
2. "Pull stakeholder needs from these interview notes [notes] and group them by persona, pain point, desired outcome, and priority."
3. "Convert this process narrative [text] into a step table with actor, action, input, output, system, and pain point."
4. "Extract open questions from this discovery document [doc] and group them by topic, decision owner, and urgency."
5. "Turn this workshop transcript [transcript] into decisions, assumptions, risks, action items, and unresolved conflicts."
6. "Extract data requirements from this business process [process] and structure them with field, definition, source, owner, and quality concern."
7. "Convert this RFP text [text] into a compliance matrix with requirement, response needed, evidence, owner, and status."
8. "Extract non-functional requirements from this document [doc] and group them by performance, security, usability, reliability, and compliance."
9. "Pull dependencies from this project brief [brief] and structure them by dependency, impacted workstream, owner, due date, and risk."
10. "Convert this stakeholder list [list] into a RACI-style table with role, responsibility, influence, decision rights, and communication needs."

## Developers

1. "Extract API endpoints from this code or documentation [source] and structure them by method, path, input, output, auth, and error behavior."
2. "Pull configuration values from this text [text] and organize them by key, value, environment, default, and security sensitivity."
3. "Convert this stack trace [trace] into structured fields: error type, failing file, line, call path, likely cause, and next investigation step."
4. "Extract TODOs and technical debt items from this code review [review] and group them by priority, owner, and module."
5. "Turn this database schema description [description] into tables, columns, keys, constraints, and relationships."
6. "Extract test cases from this feature description [description] and structure them by scenario, setup, action, expected result, and automation fit."
7. "Parse this log sample [logs] into timestamp, service, severity, request ID, event, error, and user impact."
8. "Extract breaking changes from this changelog [changelog] and structure them by affected API, migration action, and risk."
9. "Convert this implementation discussion [discussion] into decisions, open questions, tasks, risks, and owners."
10. "Extract command-line steps from this setup guide [guide] and turn them into an ordered checklist with prerequisites and validation steps."

## Senior Developers

1. "Extract architectural decisions from this RFC [rfc] and structure them by decision, rationale, alternatives, risks, and follow-up work."
2. "Pull system dependencies from this design document [doc] and organize them by service, interface, owner, SLA, and failure impact."
3. "Convert this incident report [report] into timeline, contributing factors, customer impact, detection gaps, and action items."
4. "Extract migration tasks from this plan [plan] and group them by phase, dependency, risk, rollback, and owner."
5. "Parse this architecture review [review] into findings, severity, affected area, recommendation, and decision needed."
6. "Extract operational requirements from this system description [description], including observability, alerts, runbooks, SLOs, and escalation."
7. "Convert this technical debt inventory [inventory] into themes, business impact, engineering impact, effort, and recommended sequencing."
8. "Extract capacity and performance assumptions from this proposal [proposal] and structure them with evidence, risk, and validation method."
9. "Pull cross-team dependencies from this initiative brief [brief] and structure them by team, deliverable, deadline, and coordination need."
10. "Convert this postmortem discussion [discussion] into root causes, contributing factors, lessons learned, and committed actions."

## Product People

1. "Extract customer pain points from this feedback [feedback] and group them by user segment, theme, severity, and evidence."
2. "Convert these user interviews [notes] into jobs-to-be-done, needs, blockers, quotes, and product opportunities."
3. "Pull feature requests from this feedback source [source] and structure them by request, user type, frequency, impact, and effort signal."
4. "Extract success metrics from this strategy document [doc] and separate primary metrics, secondary metrics, and guardrails."
5. "Turn this roadmap discussion [discussion] into themes, candidate bets, dependencies, risks, and unresolved decisions."
6. "Extract launch requirements from this plan [plan] and group them by product, engineering, marketing, support, sales, legal, and analytics."
7. "Convert competitor research [research] into a comparison table covering capabilities, positioning, pricing, strengths, and gaps."
8. "Extract product hypotheses from this document [doc] and structure each by assumption, expected behavior, evidence needed, and experiment idea."
9. "Pull prioritization inputs from this backlog [backlog] and structure them with RICE or WSJF fields."
10. "Convert this customer advisory board transcript [transcript] into decisions, themes, objections, opportunities, and follow-up actions."
