# Role-Based Writing and Editing Prompts

## QA (Quality Assurance)

1. "Draft a clear test summary for [feature/release], covering scope tested, major risks, defects found, and release recommendation."
2. "Rewrite this bug report [report] so it is concise, reproducible, and useful for developers."
3. "Create a polished QA status update for [stakeholders], including completed testing, blocked items, open defects, and next steps."
4. "Edit this test plan [plan] for clarity, completeness, consistent terminology, and executive readability."
5. "Draft release notes from a QA perspective for [release], highlighting known issues, validation status, and user impact."
6. "Rewrite this defect description [description] to separate expected behavior, actual behavior, steps to reproduce, and evidence."
7. "Create a concise email asking [team/stakeholder] for clarification on ambiguous requirements affecting test coverage."
8. "Edit this regression testing update [text] to make risks, timelines, and ownership easier to understand."
9. "Draft a post-test retrospective summary for [project], including what went well, what failed, and process improvements."
10. "Rewrite this QA handoff note [note] so another tester can continue the work without missing context."

## Business Analysts

1. "Draft a business requirements summary for [project] that is clear to both stakeholders and delivery teams."
2. "Rewrite this user story [story] to improve clarity, acceptance criteria, dependencies, and testability."
3. "Create a stakeholder update email for [project], covering decisions made, open questions, risks, and upcoming workshops."
4. "Edit this process documentation [doc] for consistency, plain language, and complete current-state and future-state descriptions."
5. "Draft meeting minutes from these notes [notes], including decisions, action items, owners, and deadlines."
6. "Rewrite this requirement [requirement] to remove ambiguity and make it measurable."
7. "Create a concise executive summary for this analysis [analysis], focused on business impact and recommended next steps."
8. "Edit this RFP response section [section] for persuasiveness, evidence, and alignment to the buyer's stated needs."
9. "Draft a workshop invitation for [stakeholder group] that explains purpose, preparation, agenda, and expected outcomes."
10. "Rewrite this gap analysis [analysis] so findings, implications, and recommendations are easy to scan."

## Developers

1. "Draft a pull request description for [change], including context, implementation notes, tests, risks, and rollout considerations."
2. "Rewrite this technical explanation [text] so a non-engineering stakeholder can understand the trade-offs."
3. "Create clear README documentation for [project/module], covering setup, configuration, common commands, and troubleshooting."
4. "Edit this code comment or docstring [text] for accuracy, brevity, and usefulness to future maintainers."
5. "Draft a bug fix summary for [bug], explaining root cause, fix, validation, and any remaining risk."
6. "Rewrite this API documentation [doc] to make request examples, response examples, errors, and edge cases clearer."
7. "Create a concise Slack update about [technical work], including progress, blockers, and next checkpoint."
8. "Edit this design note [note] for clearer assumptions, constraints, alternatives, and decision rationale."
9. "Draft release notes for a developer-facing change to [library/API/tool]."
10. "Rewrite this incident timeline [timeline] into a factual, neutral engineering summary."

## Senior Developers

1. "Draft an RFC for [technical decision], covering context, goals, non-goals, options, recommendation, risks, and rollout."
2. "Rewrite this architecture proposal [proposal] to make trade-offs, migration impact, and operational concerns clearer."
3. "Create an executive-friendly summary of [technical initiative] that explains business value, risk, cost, and timeline."
4. "Edit this incident postmortem [postmortem] for accuracy, accountability without blame, and actionable follow-ups."
5. "Draft a technical strategy memo for [initiative], including principles, priorities, sequencing, and success metrics."
6. "Rewrite this technical debt proposal [proposal] so it connects engineering work to customer impact and delivery risk."
7. "Create a mentoring feedback note for [engineer/situation] that is specific, balanced, and growth-oriented."
8. "Edit this cross-team announcement [announcement] for clarity on ownership, migration steps, deadlines, and support channels."
9. "Draft a risk acceptance memo for [technical risk], including mitigation, monitoring, and decision owner."
10. "Rewrite this architecture review feedback [feedback] so it is direct, constructive, and easy to act on."

## Product People

1. "Draft a PRD for [feature], covering problem, users, goals, non-goals, scope, metrics, risks, and open questions."
2. "Rewrite this product update [update] so customers understand what changed, why it matters, and what to do next."
3. "Create launch messaging for [feature] tailored to users, sales, support, and executives."
4. "Edit this roadmap narrative [text] for focus, outcome orientation, and credible sequencing."
5. "Draft a customer feedback synthesis from [source], including themes, evidence, implications, and recommended actions."
6. "Rewrite this product hypothesis [hypothesis] so it is testable and tied to measurable user behavior."
7. "Create a concise stakeholder memo for [decision], explaining options, trade-offs, recommendation, and next steps."
8. "Edit this release announcement [announcement] for clarity, tone, user benefit, and support readiness."
9. "Draft interview questions for users about [problem area], written in neutral and non-leading language."
10. "Rewrite this product strategy note [note] so it clearly connects market insight, user need, differentiation, and execution."
